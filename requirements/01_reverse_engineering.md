# 需求文档：echo_8051 网表反向工程

## 1. 项目背景

`demystify8051` 是一个芯片反向工程项目。输入为一份完全匿名化、展平的门级网表，以及对应的原始设计规格书。目标是将其还原为**结构清晰、层次分明、行为可读的 RTL 代码**，使得人类工程师能够理解和维护。

## 2. 输入件

| 文件 | 说明 |
|------|------|
| `input/echo_8051_synth.v` | Yosys 综合生成的扁平门级网表，SkyWater 130nm HD 标准单元 |
| `input/design_spec.md` | echo_8051 原始设计方案——架构图、模块分解、接口定义、SFR 映射 |

### 2.1 网表特征

| 属性 | 值 |
|------|-----|
| 工艺 | SkyWater 130nm (sky130_fd_sc_hd) |
| 总行数 | 56,136 行 |
| 标准单元实例 | 8,534 个 |
| 触发器 | 1,626 个 (dfrtp / dfstp / dfxtp) |
| 模块层次 | 无——单层 `echo_8051_top` |
| 连线命名 | 匿名化 `_00000_` ~ `_XXXXX_` |
| 顶层端口 | clk, rst_n, int0_n, int1_n, p0-p3, rxd, txd, ale, psen_n, rd_n, wr_n, ea_n |

## 3. 目标产出

### 3.1 RTL 代码

将扁平网表还原为**带行为语义的分层 RTL**：

```
rtl/
├── echo_8051_top.v        # 顶层集成
├── cpu_core.v              # CPU 核
│   ├── alu.v               #   8 位 ALU
│   ├── decoder.v           #   指令译码器
│   ├── control_fsm.v       #   控制状态机
│   ├── reg_file.v          #   寄存器组 (4 banks × 8)
│   └── psw.v               #   程序状态字
├── iram.v                  # 128B 内部 RAM
├── sfr_block.v             # SFR 寄存器组
├── prom.v                  # 4KB 程序 ROM
├── timer.v                 # T0/T1 定时计数器
├── uart.v                  # 串行口
├── intc.v                  # 中断控制器
└── io_ports.v              # P0-P3 I/O
```

## 4. 技术路线

### 4.1 整体策略：Spec-Driven + Netlist-Driven 双向推进

**自顶向下（Spec-Driven）**：
- 从 `design_spec.md` 中提取架构信息——模块列表、接口定义、状态机描述、SFR 地址映射
- 在网表中定位与各模块对应的子电路区域（通过端口信号追踪、寄存器聚类、控制信号模式匹配）

**自底向上（Netlist-Driven）**：
- 将扁平网表中的 1,626 个触发器聚类为功能寄存器组
- 从触发器出发：向前追踪数据通路（组合逻辑锥），向后追踪控制信号
- 识别 8,534 个标准单元组成的逻辑锥的功能（通过布尔函数抽取、真值表分析）
- 从顶级 I/O 端口反向追踪，逐级识别子模块边界

### 4.2 分阶段执行

#### Phase 0 — 网表解析与基础设施（准备阶段）

- 解析网表，建立内部数据结构（cell graph / netlist graph）
- 识别所有 1,626 个寄存器的 fan-in / fan-out 锥
- 识别时钟域和复位信号
- 将匿名连线重新分组，按功能簇聚类

#### Phase 1 — 顶层结构识别

- 从顶层 14 个端口开始追踪，识别 I/O 路径
- 通过寄存器簇 + 输入输出端口关联，定位各子模块的大致区域
- 建立 rough hierarchy——将 8,534 个 cell 划分为功能区域

#### Phase 2 — 逐模块反向

按优先级顺序：

| 优先级 | 模块 | 识别线索 |
|--------|------|---------|
| P0 | ALU | 8 位数据通路、运算控制信号模式 |
| P0 | 寄存器组 | 32×8 寄存器阵列、RS1/RS0 选择 |
| P0 | 译码器 | 从指令寄存器出发的 255→控制信号映射 |
| P0 | 控制状态机 | 状态触发器 + 微码 ROM |
| P1 | 内部 RAM | 128×8 存储阵列 |
| P1 | SFR 块 | 寄存器 → 地址解码器的映射 |
| P2 | 定时器 T0/T1 | 16 位计数器 + 控制寄存器 |
| P2 | 中断控制器 | 优先级编码器 + 向量生成 |
| P2 | 串行口 UART | 移位寄存器 + 波特率生成 |
| P3 | I/O 端口 | 端口寄存器 + 复用逻辑 |

#### Phase 3 — RTL 编写与集成

- 每个模块编写清晰的行为级 RTL（不要逐门复制！）
- 保持与设计规格的接口一致性
- 顶层集成，验证模块间连线

#### Phase 4 — 验证闭环

- 模块级 testbench 验证每个子模块
- 系统级：运行 8051 机器码，比对 Python/C++ ISS 参考模型
- 目标：指令覆盖 255/255 操作码

## 5. 质量要求

| 指标 | 目标 |
|------|------|
| RTL 可综合性 | Yosys 综合 zero errors |
| 功能等价性 | 与原始网表在所有 255 操作码上行为一致 |
| RTL 代码风格 | 可读、有注释、模块化、非 obfuscated |
| 文档完整性 | 每个识别决策有记录、关键信号命名有语义 |

## 6. 工具链

| 用途 | 工具 | 说明 |
|------|------|------|
| 网表解析 | Python (自定义脚本) | 解析 Verilog 网表、构建 cell graph |
| 逻辑分析 | Python + Yosys | 布尔函数抽取、等价性检查 |
| RTL 仿真 | Icarus Verilog / Verilator | 仿真验证 |
| 综合验证 | Yosys | 确认 RTL 可综合 |
| 参考模型 | Python/C++ ISS (已有) | 黄金参考模型比对 |

## 7. 非目标（本项目不做）

- 不做逐门（gate-by-gate）的 1:1 复制——那和网表没区别
- 不做物理验证（DRC/LVS）——那是原项目 Phase 5 的范围
- 不追求时序优化——行为等价优先
- 不逆向 ROM 内容——ROM 是行为模型，直接使用
